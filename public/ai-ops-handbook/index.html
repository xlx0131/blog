第1章 Linux操作系统基础

1
运维基本导论与Linux操作系统入门
达标
详细讲解

运维（Operations）是IT领域三大核心职能之一（产品/研发/运维）。运维工程师的职责可以概括为三个关键词：监控、故障处理、自动化。

为什么要学运维？ 任何线上服务都需要7×24小时稳定运行。没有运维，系统出故障时就没有人第一时间发现和修复；没有运维，新代码就无法安全地部署到生产环境。运维是保障业务连续性的最后一道防线。

云计算定义与优势：通过互联网按需提供计算资源。核心优势：按需自助、广泛网络接入、资源池化、快速弹性、可计量服务。

服务器分类：塔式（小型企业）、机架式（数据中心标准19英寸机架）、刀片（密度最高，共享电源散热）。

硬件组成：主板→CPU（大脑）→内存RAM（临时存储，断电丢失）→硬盘（HDD便宜容量大/SSD快贵）→网卡→电源（双冗余）。

企业IT角色：产品经理（定义需求）→研发（编码）→测试（验证Bug）→运维（部署监控故障处理）。

环境分类：开发dev→测试test→预发布staging→生产prod（面向真实用户，操作需谨慎）。

网络设备：HUB（已淘汰）、交换机（L2，独享带宽）、路由器（L3，跨网段通信）。介质：双绞线（100米以内）、光纤（远距离高速）。

通信三要素：IP（地址，唯一标识）、端口（服务标识，0-65535）、协议（通信规则，TCP可靠/UDP快速）。

CentOS Stream vs Ubuntu双系统：CentOS Stream滚动更新（Fedora↔RHEL之间），Ubuntu社区活跃文档齐全，双系统通过GRUB引导菜单选择。

如何用
# 查看CPU信息
cat /proc/cpuinfo | grep "model name" | head -1
# 查看内存/磁盘
free -h && lsblk
# 查看网卡
ip a
# 判断环境：通过主机名(prod-xxx)、IP段(10.x内网)区分

目录命令（pwd/ls/cd/mkdir）
达标
详细讲解

Linux文件系统是倒挂树形结构，根目录/是一切起点。

pwd显示当前绝对路径，脚本中判断位置时必备。

ls -lah是最常用组合：-l详细信息（权限/所有者/大小/时间），-a含隐藏文件（.开头），-h人类可读大小（1K/234M），-t按时间排序。

cd切换目录：cd ~回家目录，cd -回上一个目录，cd ..回上级。

mkdir -p递归创建多级目录，如mkdir -p /data/app/logs一次性创建所有父目录。

如何用
# 创建项目目录结构
mkdir -p /opt/myapp/{bin,conf,logs,data}
# 脚本中判断目录是否存在
[ -d "/opt/myapp" ] || mkdir -p /opt/myapp/logs

文件操作（touch/cp/mv/rm）
达标
详细讲解

touch创建空文件或更新时间戳（触发监控脚本/日志轮转时有用）。

cp -a归档模式复制，保留权限/时间戳/软链接等所有属性，备份配置文件时必备。

mv移动=重命名（同一目录）或剪切（不同目录）。

rm最危险命令之一。-r递归，-f强制。生产环境绝不能用rm -rf /或rm -rf /*。建议先mv到回收站再确认删除。

如何用
# 备份配置（保留权限和时间戳）
cp -a /etc/nginx/nginx.conf /etc/nginx/nginx.conf.bak.$(date +%Y%m%d)
# 批量重命名 .txt → .bak
for f in *.txt; do mv "$f" "${f%.txt}.bak"; done

文本编辑（vi/vim三种模式）
达标
详细讲解

为什么要学Vim？ SSH远程连接服务器后唯一保证可用的编辑器，无需图形界面。

命令模式（默认）：dd删行、yy复制、p粘贴、u撤销、/关键词搜索、gg文件头、G文件尾。

插入模式：i当前位置插入、a后插入、o下行插入、Esc返回命令模式。

末行模式：:w保存、:q退出、:wq保存退出、:q!强制退出、:%s/旧/新/g全文替换。

如何用
vim /etc/nginx/nginx.conf
# 批量注释：Ctrl+v→选多行→Shift+i→输#→Esc
# 全文替换：:%s/80/8080/g

文件查看（cat/tail/less）
达标
详细讲解

cat一次性输出全部内容，适合小文件（-n加行号）。

tail -f运维最常用命令之一：实时跟踪日志文件末尾新增内容，Ctrl+C退出。-F比-f更健壮（文件被删除旋转后自动重新打开）。-n 100显示最后100行。

less分页查看大文件：空格下一页、b上一页、/关键词搜索、q退出。比more强大（可向前翻页）。

如何用
# 实时监控nginx访问日志
tail -f /var/log/nginx/access.log
# 查看包含ERROR的最后50行
grep "ERROR" /var/log/app.log | tail -50

解压缩（tar/zip）
达标
详细讲解

tar Linux最常用归档工具。记忆法：c=create、x=extract、v=verbose、f=file（必须最后）、z=gzip(.tar.gz)、j=bzip2(.tar.bz2更高压缩率但更慢)。最常用：tar -czvf打包压缩、tar -xzvf解压。

zip/unzip跨平台兼容好，Windows/macOS原生支持。zip -r递归压缩目录，unzip -d 目标目录解压到指定位置。

如何用
# 打包压缩
tar -czvf myproject.tar.gz /opt/myproject/
# 解压到指定目录
tar -xzvf myproject.tar.gz -C /opt/
# 不解压查看内容
tar -tzvf myproject.tar.gz

包管理（dnf与rpm）
达标
详细讲解

dnf是yum的下一代版本，自动处理依赖关系。dnf install 包名安装、dnf remove 包名删除、dnf update更新全部、dnf search 关键词搜索、dnf info 包名查看详情。

rpm底层包管理，直接操作.rpm文件。rpm -ivh 包.rpm安装（不自动处理依赖）、rpm -e 包名卸载、rpm -qa列出所有已安装包、rpm -ql 包名查看包安装了哪些文件、rpm -qf /路径/文件反向查文件属于哪个包。

如何用
dnf install -y nginx
dnf search mysql
rpm -qa | grep kernel
rpm -qf /etc/ssh/sshd_config   # 查文件属于哪个包

服务管理（systemctl）
达标
详细讲解

systemd是现代Linux的初始化系统和服务管理器，替代传统SysV init。核心命令：systemctl start/stop/restart 服务名启停重启、enable/disable开机自启设置、status查看状态（是否运行/PID/最近日志）、is-active检查是否运行中、list-units --type=service列出所有服务。

journalctl -u 服务名 -f实时查看该服务的日志。

如何用
systemctl start nginx && systemctl enable nginx
systemctl status nginx
journalctl -u nginx -f    # 实时查看nginx日志

定时任务（crontab）
达标
详细讲解

cron守护进程每分钟检查crontab文件，匹配即执行。语法：分 时 日 月 周 命令（0-59 0-23 1-31 1-12 0-7）。*任意、*/5每5单位、1,3,5枚举。

crontab -e编辑、-l列出、-r删除。建议任务输出重定向到日志文件>> /var/log/xxx.log 2>&1。

如何用
# 每天凌晨2点备份
0 2 * * * /opt/scripts/backup.sh >> /var/log/backup.log 2>&1
# 每5分钟健康检查
*/5 * * * * /opt/scripts/health_check.sh

防火墙管理（firewalld/ufw）
达标
详细讲解

防火墙是服务器安全第一道防线，精确控制哪些端口对外暴露。

firewalld（RHEL/CentOS系）：基于zone+service管理。firewall-cmd --add-port=80/tcp --permanent永久开放、--reload重载生效。注意--permanent只写配置文件不立即生效。

ufw（Ubuntu/Debian系）：语法更简洁。ufw allow 80/tcp开放、ufw enable启用。

如何用
# firewalld
firewall-cmd --add-service=http --permanent
firewall-cmd --add-port=8080/tcp --permanent
firewall-cmd --reload
# ufw
ufw allow 22/tcp && ufw allow 80/tcp && ufw enable
11
进程管理（ps/kill/ss）
达标
详细讲解

ps aux查看所有进程快照。关键字段：USER（所有者）、PID（进程ID）、%CPU、%MEM、STAT（R运行/S睡眠/Z僵尸）。ps aux --sort=-%mem | head -6查内存TOP5。

kill -15 PID优雅终止（SIGTERM，给进程清理机会），kill -9 PID强制杀死（SIGKILL，无法忽略），kill -1 PID重载配置（SIGHUP）。优先用-15。

ss -tlnp查看TCP监听端口及进程（netstat现代替代品，速度更快）。ss -tlnp | grep 8080查谁占用了8080端口。

如何用
ss -tlnp | grep 8080        # 查端口占用
ps aux | grep nginx          # 查nginx进程
kill -1 $(cat /var/run/nginx.pid)  # 优雅重载nginx

第2章 云平台ECS服务器与云监控

1
云服务器定义、厂商、特点、优势
达标
详细讲解

云服务器ECS（Elastic Compute Service）本质上是一台运行在云厂商数据中心里的虚拟机。你通过互联网远程管理它，就像管理一台真实的物理服务器一样，但不需要关心硬件采购、机房网络、电力空调等底层基础设施。

主要云厂商：国内以阿里云（市场份额第一）、腾讯云、华为云为主；国际以AWS（亚马逊）、Azure（微软）、GCP（谷歌）为主。各家提供的核心服务类似（计算/存储/网络/数据库），但API和产品名称不同。

核心特点：虚拟化（一台物理机运行多台虚拟机，共享硬件降低成本）、按需分配（需要多少买多少，随时扩容缩容）、弹性伸缩（业务高峰期自动增加服务器，低谷期自动减少）、即开即用（几分钟内创建一台服务器，不用等采购流程）。

优势：相比自建机房，云服务器免去了硬件采购（通常2-4周）、机房托管、网络带宽申请等繁琐流程；按量付费（用多少付多少，不用不花钱）比一次性买断硬件更灵活；全球多地域部署（一键在东京/新加坡/法兰克福开服，自建机房做不到）。

如何用
# 阿里云CLI创建ECS实例
aliyun ecs CreateInstance --RegionId cn-hangzhou \
  --InstanceType ecs.g7.large --ImageId centos_stream_9

阿里云ECS实例选择与购买流程
达标
详细讲解

实例规格选择：经济型e（最便宜，适合测试学习）、通用算力型u（性价比之选，适合Web应用）、计算型c（CPU密集场景如批量处理）、内存型r（大内存需求如数据库/缓存）。规格名如ecs.g7.large中g7代表第7代通用型、large代表2vCPU+8G内存。

计费方式：按量付费（按小时/秒计费，随时释放不花钱，适合测试学习）、包年包月（预付打折，适合长期运行的正式业务）、抢占式实例（最高1折但可能被回收，适合无状态批处理任务）。

购买流程：选择地域（离用户越近延迟越低）→ 选择可用区（同城不同机房，做高可用用）→ 选择实例规格 → 选择镜像（CentOS/Ubuntu/Windows等）→ 配置网络（VPC虚拟私有网络/安全组防火墙）→ 配置存储（系统盘+数据盘）→ 设置密钥对或密码 → 确认订单付款。

如何用
  • 学习阶段选"按量付费+最低规格（1vCPU 0.5G内存）"，每天几毛钱。
  • 安全组创建规则：只开放必要端口（22 SSH / 80 HTTP / 443 HTTPS），来源IP设为0.0.0.0/0或限定自己的IP。

ECS使用：重启/资源调整/安全组/远程连接
达标
详细讲解

重启：控制台点"重启"按钮或SSH登录后执行reboot。注意重启会导致短暂服务中断。

资源调整（升降配）：云服务器最方便的功能之一——随时调整CPU核数、内存大小、带宽上限，不需要重装系统（但通常需要先关机）。

安全组：相当于云上的虚拟防火墙。入方向（Ingress）控制谁可以访问你的ECS；出方向（Egress）控制ECS可以访问谁。规则由协议类型、端口范围、源IP/安全组组成。

远程连接：SSH（Linux）是最常用方式：ssh root@公网IP。或者使用控制台自带的VNC远程连接（不需要公网IP也能连，适合网络出问题时紧急排查）。

如何用
# SSH远程连接（使用密钥对）
ssh -i ~/.ssh/mykey.pem root@123.123.123.123
# 安全组规则示例（仅开放SSH给指定IP）
# 控制台操作：入方向 → 添加规则 → 协议TCP 端口22 源IP:你的IP/32

云监控：介绍/开通/主机监控/报警/可视化大屏
达标
详细讲解

云监控（CloudMonitor）是阿里云提供的免费监控服务。它能自动采集ECS的CPU使用率、内存使用率、磁盘IO、网络流量等指标，无需安装额外Agent（部分指标需要安装云监控插件）。

主机监控：默认展示CPU使用率、内存使用率、磁盘使用率、公网带宽等核心指标。你可以设置Dashboard仪表盘，把多台服务器的关键指标放在同一个页面查看。

资源报警：设置阈值触发告警通知——如"CPU使用率>90%持续5分钟"就发送短信/邮件/钉钉通知。报警规则包含：监控项（如CPU）、统计周期（1分钟/5分钟）、统计方法（平均值/最大值）、阈值、触发次数、沉默时间（避免重复告警轰炸）。

可视化大屏：将多个监控图表组合成全屏Dashboard，适合投屏到办公室大电视实时展示业务健康状态。

如何用
# 典型报警规则配置
# 监控项：CPU使用率
# 统计周期：5分钟  统计方法：平均值
# 阈值：>= 90%  触发次数：连续3次
# 通知方式：短信+钉钉机器人
# 沉默时间：30分钟（避免重复骚扰）

用户与组基本概念、用户组操作、用户操作
达标
详细讲解

Linux是多用户操作系统，一台服务器可以同时被多个用户登录使用。用户管理是运维安全的基础——每个应用、每个运维人员都应该有独立账号，严禁所有人共用root。

用户(UID)：Linux用数字UID标识用户（root=0，系统用户1-999，普通用户≥1000）。/etc/passwd存储用户信息。

组(GID)：用户属于一个主组(GID)和多个附加组。组的存在让权限管理更高效——把多个用户加到一个组，统一授权。/etc/group存储组信息。

核心命令：useradd 用户名创建用户（自动创建家目录/home/用户名、设置默认shell为/bin/bash、分配UID），-m强制创建家目录，-s /sbin/nologin创建不能登录的系统账号。创建用户后需passwd 用户名设置密码。usermod -aG 组名 用户名将用户加入附加组（必须加-a否则会覆盖原有组）。userdel -r 用户名删除用户及家目录。

groupadd 组名创建组，groupdel 组名删除组。id 用户名查看用户的UID/GID和所属组。

如何用
# 创建应用专用账号（不能登录）
useradd -s /sbin/nologin -M nginx
# 创建运维人员账号并加入wheel组（sudo权限组）
useradd -m devops
passwd devops
usermod -aG wheel devops
# 验证
id devops
# 查看所有用户
cat /etc/passwd

权限管理、umask默认权限、特殊权限
达标
详细讲解

Linux权限模型：每个文件/目录有三组权限——所有者(u)、所属组(g)、其他人(o)，每组有读(r=4)、写(w=2)、执行(x=1)三种权限。chmod 755 file数字法：所有者rwx(7)、组r-x(5)、其他人r-x(5)。chmod u+x file符号法：给所有者加执行权限。

umask：定义新建文件/目录的默认权限。默认值022，新建文件权限=666-022=644（rw-r--r--），新建目录权限=777-022=755（rwxr-xr-x）。临时修改：umask 027，永久修改：写入/etc/profile或~/.bashrc。

特殊权限：SUID(冒险位S)：如果可执行文件设置了SUID，任何用户执行它时都以文件所有者的身份运行（如/usr/bin/passwd，普通用户修改密码需要写/etc/shadow，但该文件仅root可写——passwd命令设置了SUID，执行时临时获得root权限）。SGID：类似SUID但是以组身份运行。粘滞位(Sticky bit, T)：只有目录有意义（如/tmp），设置后用户只能删除自己的文件，不能删除别人的文件。ls -l /tmp看到drwxrwxrwt最后的t就是粘滞位。

如何用
# 设置权限
chmod 755 /opt/app/bin/start.sh
chmod 600 /opt/app/conf/secret.key  # 仅所有者可读写
# 设置SUID
chmod u+s /usr/local/bin/mytool
# 粘滞位（共享目录）
chmod +t /data/shared
# 递归修改所有者
chown -R appuser:appgroup /opt/app/

ACL权限控制
良好
详细讲解

ACL（Access Control List）是传统rwx权限模型的扩展。传统模型只能为一个所有者+一个组+其他人设权限，ACL允许为任意多个用户/组设置独立权限。

为什么需要ACL？ 例如：/data/project目录，开发者组需要读写、测试组只需读、项目经理需要读、运维需要完全控制——传统rwx模型只能设一个组权限，无法满足。ACL完美解决这个问题。

setfacl -m u:用户名:权限 文件设置用户ACL，setfacl -m g:组名:权限 文件设置组ACL。getfacl 文件查看ACL。ACL权限在ls -l中显示为+号（如-rwxrwxr-x+）。

如何用
# 给devops用户读权限
setfacl -m u:devops:r /data/project/config.ini
# 给dev组读写权限
setfacl -m g:dev:rw /data/project/
# 查看ACL
getfacl /data/project/
# 删除某条ACL
setfacl -x u:devops /data/project/
# 删除所有ACL
setfacl -b /data/project/

SSH服务与DNS服务
达标
详细讲解

SSH（Secure Shell）是运维的"瑞士军刀"。它本质是加密的远程登录协议，替代了明文的telnet。SSH工作原理：客户端发起连接→服务端发送公钥→双方协商加密算法→建立加密通道→验证用户身份（密码或密钥对）→建立会话。

SSH密钥对登录比密码登录更安全：①私钥在本地，密码从不经过网络传输；②私钥可以加密码短语保护；③一台服务器的authorized_keys可以放多个公钥（多人管理）。配置步骤：ssh-keygen -t rsa生成密钥对→ssh-copy-id user@host把公钥复制到远程服务器→直接ssh user@host免密登录。

DNS（Domain Name System）是互联网的"电话簿"，把域名（www.baidu.com）翻译成IP地址（110.242.68.66）。DNS是分层结构：根域名服务器(.)→顶级域名(.com)→权威域名服务器(baidu.com)→具体记录。运维需要掌握：A记录（域名→IPv4）、AAAA（域名→IPv6）、CNAME（别名，如www→主域名）、MX（邮件服务器）、NS（域名服务器）。

BIND是最流行的开源DNS服务器软件。配置内外网分离解析：内网用户查询返回内网IP（速度快不走公网），外网用户查询返回公网IP。

如何用
# 生成SSH密钥对
ssh-keygen -t rsa -b 4096 -C "devops@company.com"
# 免密登录配置
ssh-copy-id root@192.168.1.100
# SCP免密传输
scp -r /local/dir root@192.168.1.100:/remote/dir
# BIND安装
dnf install -y bind bind-utils
# 测试DNS解析
nslookup www.baidu.com
dig @8.8.8.8 www.example.com

数据同步与FTP服务
达标
详细讲解

Rsync是Linux下最强大的文件同步工具。它的核心优势是增量传输——只传输变化的文件部分，不是整个文件重传。这对运维非常重要：备份几百G的数据，如果每次都全量拷贝会浪费大量带宽和时间。

Rsync常用选项：-a归档模式（保留所有属性），-v详细输出，-z传输时压缩，--delete删除目标端多余文件（保持完全一致），--exclude排除指定文件。结合cron可以设置定时同步。

Inotify是Linux内核提供的文件系统事件监控机制。Inotify+Rsync可以实现实时同步——当源目录有文件变化时，自动触发Rsync同步。相比定时同步（可能延迟几分钟），实时同步几乎是秒级。

FTP（File Transfer Protocol）：最古老的文件传输协议。主动模式（PORT）：服务器主动连接客户端的数据端口（客户端防火墙可能拦截）。被动模式（PASV）：客户端连接服务器的数据端口（更防火墙友好，推荐）。vsftpd是最安全快速的Linux FTP服务器。注意FTP是明文协议，生产环境建议用SFTP（基于SSH加密）或FTPS（FTP over TLS）。

如何用
# Rsync定时同步
rsync -avz /data/source/ /data/backup/
# Rsync远程同步
rsync -avz -e ssh /local/ user@remote:/backup/
# 定时同步到crontab
0 */4 * * * rsync -avz --delete /data/ backup@backup-server::data
# vsftpd安装
dnf install -y vsftpd
systemctl start vsftpd && systemctl enable vsftpd

NFS与NTP时间同步服务
达标
详细讲解

NFS（Network File System）：让多台服务器共享同一个目录，像访问本地文件一样访问远程文件。典型场景：多台Web服务器共享上传目录（用户上传的图片存在NFS上，所有Web服务器都能读取）。

Chrony是NTP（Network Time Protocol）的现代实现，比传统ntpd更快更精准。为什么时间同步对运维这么重要？①分布式系统中服务器时间不一致会导致日志排查困难（A服务器日志说"10:05出错"，B服务器日志说"10:03出问题"，但你不知道谁的时间不对）；②数据库主从复制依赖时间戳；③安全认证协议（如Kerberos）要求时间差不超过5分钟。Chrony配置很简单：/etc/chrony.conf中指定上游时间服务器，如server ntp.aliyun.com iburst。

如何用
# NFS服务端
dnf install -y nfs-utils
echo "/data/shared 192.168.1.0/24(rw,sync,no_root_squash)" >> /etc/exports
systemctl start nfs-server
# NFS客户端
mount -t nfs 192.168.1.10:/data/shared /mnt/shared
# 写入/etc/fstab实现开机自动挂载
# Chrony时间同步
dnf install -y chrony
systemctl start chronyd && systemctl enable chronyd
chronyc sources -v  # 查看时间源状态

日志管理与深层防火墙
达标
详细讲解

系统日志是故障排查的第一手资料。/var/log/messages（通用系统日志）、/var/log/secure（认证安全日志，SSH登录记录）、/var/log/cron（定时任务日志）、/var/log/nginx/（Web服务日志）。RSYSLOG是Linux默认的日志服务，支持日志远程转发（多台服务器日志集中到一台日志服务器）。

logrotate日志轮替：防止日志文件无限增长占满磁盘。按天/周/月轮替，保留最近N份，旧日志自动压缩删除。/etc/logrotate.conf是全局配置，/etc/logrotate.d/下是各服务的单独配置。

SELinux（Security-Enhanced Linux）是NSA开发的强制访问控制系统。它在传统权限之上加了一层安全上下文（context），即使文件权限是777，SELinux仍然可以阻止访问。很多运维初学者遇到"权限正确但访问被拒"的问题就是SELinux导致的。生产环境建议保持enforcing模式（不要直接关掉），而是学习如何配置正确的上下文。

如何用

磁盘管理：MBR/GPT分区、LVM逻辑卷
达标
详细讲解

磁盘结构：物理硬盘→分区→格式化（创建文件系统）→挂载到目录→才能使用。

MBR vs GPT：MBR（Master Boot Record）传统分区表，最大支持2TB磁盘和4个主分区（需扩展分区来解决4主分区限制）。GPT（GUID Partition Table）新一代分区表，支持超过2TB磁盘和128个分区，使用UEFI启动。现在新装系统都推荐GPT。

文件系统：ext4（Linux传统，稳定成熟）、XFS（RHEL/CentOS默认，支持更大文件和更好并发性能）、Btrfs（新一代，支持快照和压缩）。

LVM（Logical Volume Manager）：在物理磁盘和文件系统之间加了一层抽象。核心概念：PV（物理卷，实际的磁盘分区）→ VG（卷组，多个PV组成的存储池）→ LV（逻辑卷，从VG中划分出来的虚拟分区）。LVM最大的好处是动态扩容——数据库的LV空间不够了，加一块硬盘做成PV加入VG，然后在线扩展LV，不需要重启服务！没有LVM的话，磁盘扩容需要停机、备份数据、重新分区、恢复数据，影响业务。

如何用
# 查看磁盘
lsblk && fdisk -l
# 创建LVM（完整流程）
pvcreate /dev/sdb              # 创建PV
vgcreate vg_data /dev/sdb      # 创建VG
lvcreate -L 100G -n lv_mysql vg_data  # 创建LV
mkfs.xfs /dev/vg_data/lv_mysql        # 格式化
mount /dev/vg_data/lv_mysql /data/mysql  # 挂载
# 在线扩容（LV还有剩余空间时）
lvextend -L +50G /dev/vg_data/lv_mysql
xfs_growfs /data/mysql

1
网络设备与TCP/IP模型
达标
详细讲解

L2交换机：工作在数据链路层，基于MAC地址转发。同一VLAN内的设备通过交换机通信。L2交换机能学习MAC地址和端口的对应关系（MAC地址表），实现精确转发而非广播。

L3交换机：具有路由功能的交换机，既能做VLAN间路由，又能做二层转发。比"路由器+交换机"组合更高效（路由转发在硬件层面完成）。

防火墙：控制网络流量的安全设备。下一代防火墙（NGFW）不仅根据IP/端口过滤，还能识别应用（如能区分"HTTP流量"和"微信流量"）。负载均衡器：将流量分发到多台后端服务器，避免单点过载。

TCP/IP四层模型（自下而上）：网络接口层（物理介质/网卡驱动/MAC地址）→ 网络层（IP协议/路由器/ICMP）→ 传输层（TCP可靠传输/UDP快速传输）→ 应用层（HTTP/SSH/DNS等具体应用协议）。理解分层模型是网络排错的基础：ping通说明网络层以下正常；telnet IP port通说明传输层正常；curl通说明应用层正常。

IP地址与子网掩码：IP地址分为网络部分和主机部分，子网掩码标明分界线。如192.168.1.100/24，网络部分是192.168.1，主机部分是100，/24表示子网掩码255.255.255.0（前24位是网络位）。同一个网络的设备可以直接通信，不同网络的设备需要路由器转发。

如何用

# 查看路由表

# 分层排错

telnet 192.168.1.100 3306      # 传输层（MySQL端口）
curl http://192.168.1.100:8080 # 应用层

交换机与VLAN、在线实验平台（eNSP/EVPN）
达标
详细讲解

VLAN（Virtual LAN）：在一台物理交换机上划分多个逻辑隔离的广播域。没有VLAN时，整个交换机的所有端口在同一个广播域，任意主机发广播包所有主机都收到。划分VLAN后，VLAN10的主机和VLAN20的主机在逻辑上是隔离的，不能直接通信。

为什么要用VLAN？ ①安全隔离（财务部门VLAN10不能直接访问研发部门VLAN20）；②减少广播风暴（广播包被限制在VLAN内）；③灵活的管理（物理位置分散的同事可以属于同一个VLAN）。

eNSP（华为）和EVE-NG是网络模拟器，让你在电脑上搭建虚拟网络拓扑，不用买真实的交换机路由器就能学习VLAN、路由、NAT等网络技术。

路由基础、VLAN间路由、NAT、ACL
达标
详细讲解

路由：决定数据包从源到目的地的路径。路由器根据路由表（目标网络→下一跳/出接口）转发数据。静态路由：手动配置，简单可靠但无法自动适应拓扑变化。动态路由（OSPF/BGP等）：路由器之间自动交换路由信息，适合大型复杂网络。

VLAN间路由：不同VLAN的主机默认不能通信，需要通过L3设备（路由器或L3交换机）做VLAN间路由。有两种方式：①单臂路由（一个物理接口划分多个子接口，每个子接口对应一个VLAN的网关）②L3交换机SVI（Switch Virtual Interface，给每个VLAN配置一个IP作为该VLAN的网关）。

NAT（Network Address Translation）：网络地址转换，解决IPv4地址不够用的问题。内网使用私有IP（192.168.x.x/10.x.x.x/172.16-31.x.x），通过NAT路由器转换成公网IP访问互联网。SNAT（源地址转换）让内网上网；DNAT（目标地址转换）让外网访问内网服务器（端口映射）。

ACL（Access Control List）：访问控制列表，在路由器/防火墙上过滤流量。规则按顺序匹配，命中即执行（放行或拒绝），最后默认拒绝。ACL可以基于源IP、目标IP、协议、端口组合条件。

如何用

# Linux 实现SNAT（iptables）

# 端口映射（DNAT）
iptables -t nat -A PREROUTING -p tcp --dport 80 -j DNAT --to 192.168.1.10:8080

达标
详细讲解

Wireshark抓包分析：运维最重要的网络排错工具。它捕获经过网卡的所有数据包，让你看到网络中到底发生了什么。过滤语法最常用：tcp.port==80只看HTTP流量，ip.addr==192.168.1.100只看某IP，http.request.method==GET只看HTTP GET请求。通过"Follow TCP Stream"可以还原完整的TCP会话。典型场景：用户说"网站打不开"，抓包发现TCP三次握手失败→检查防火墙；抓包发现HTTP返回500→检查应用日志。

故障排查方法：ping测试连通性（网络层）、traceroute追踪路径（看数据包经过哪些路由器）、nslookup/dig排查DNS问题、telnet IP port测试端口是否开放。

# 追踪路由
traceroute 8.8.8.8
# 测试端口

第1章 Nginx网站服务

1
Nginx安装、基本配置、模块化结构
达标
详细讲解

Nginx是俄罗斯人Igor Sysoev开发的高性能Web服务器，现在是全球使用量最大的Web服务器（超过Apache）。它的核心优势是异步事件驱动架构——一个worker进程可以同时处理数千个并发连接（Apache是每个连接一个进程/线程，并发高时内存爆炸）。

为什么Nginx这么重要？ 几乎所有互联网公司的Web层都用Nginx：作为静态资源服务器（图片/CSS/JS直接返回，速度快）、作为反向代理（转发动态请求到后端应用服务器如Tomcat/Node.js/Python）、作为负载均衡器（分发流量到多台后端服务器）、作为SSL终端（统一处理HTTPS加解密，减轻后端压力）。

模块化设计：Nginx核心很小，功能通过模块实现。nginx -V查看编译了哪些模块。核心模块（http_core处理HTTP请求）、标准模块（ngx_http_ssl_module处理HTTPS、ngx_http_proxy_module反向代理）、第三方模块（ngx_http_lua_module用Lua扩展Nginx）。

配置文件结构：/etc/nginx/nginx.conf主配置文件→http{}块里include /etc/nginx/conf.d/*.conf按站点拆分配置→每个server{}块定义一个虚拟主机（域名绑定）。

如何用
# 安装Nginx
dnf install -y nginx
# 查看版本和模块
nginx -V 2>&1 | tr ' ' '\n' | grep module
# 配置文件语法检查（修改配置后必做！）
nginx -t
# 启动/重载
systemctl start nginx
nginx -s reload   # 优雅重载（不中断现有连接）

虚拟主机、root与alias、location规则
达标
详细讲解

虚拟主机：一台物理服务器上运行多个网站（通过不同域名区分）。原理：HTTP请求头中有Host: www.example.com，Nginx根据Host头匹配对应的server块。

root vs alias（高频面试题）：root是把location路径追加到root路径后面，alias是替换。例如location /app/ { root /var/www; }意味着访问/app/index.html→实际文件/var/www/app/index.html。而location /app/ { alias /var/www/myapp/; }意味着访问/app/index.html→实际文件/var/www/myapp/index.html（location的/app被alias替换了）。

location匹配优先级（从高到低）：①=精确匹配（如= /favicon.ico）②^~前缀匹配（匹配后不再检查正则）③~或~*正则匹配（~区分大小写，~*不区分）④普通前缀匹配（最长匹配优先）。理解优先级是配置Nginx的必修课。

如何用
# 虚拟主机配置
server {
    listen 80;
    server_name www.example.com;
    root /var/www/example;
}
server {
    listen 80;
    server_name blog.example.com;
    root /var/www/blog;
}
# location优先级示例
location = /health { return 200 "OK"; }          # 最高优先级
location ^~ /static/ { root /data; }              # 次高
location ~ \.php$ { fastcgi_pass ...; }           # 正则
location / { root /var/www; }                     # 兜底

反向代理与SSL证书部署
达标
详细讲解

反向代理：客户端→Nginx（反向代理）→后端服务器。客户端只知道Nginx的地址，不知道后端服务器的存在。反向代理的好处：①安全（隐藏后端真实IP）、②负载均衡（一台Nginx后面接多台应用服务器）、③SSL卸载（Nginx处理HTTPS，后端用HTTP减轻加解密负担）、④缓存加速（Nginx缓存静态内容直接返回）。

proxy_pass配置反向代理核心指令。proxy_set_header转发客户端真实信息（Host头、真实IP X-Real-IP、代理链 X-Forwarded-For），否则后端只能看到Nginx的IP。

SSL/TLS证书：HTTPS的加密基础。证书类型：①DV（域名验证，免费Let's Encrypt，适合个人网站）②OV（组织验证，适合企业官网）③EV（扩展验证，地址栏显示绿色公司名，适合金融/电商）。Let's Encrypt + Certbot免费自动签发续期（90天有效，自动续期），是中小项目首选。

如何用
# 反向代理配置
location /api/ {
    proxy_pass http://backend-cluster;
    proxy_set_header Host $host;
    proxy_set_header X-Real-IP $remote_addr;
    proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
}
# Certbot免费SSL证书
dnf install -y certbot python3-certbot-nginx
certbot --nginx -d example.com -d www.example.com
# 配置自动续期（crontab）
0 3 * * * certbot renew --quiet

负载均衡算法、高可用Keepalived
达标
详细讲解

upstream模块定义后端服务器组。负载均衡算法：①轮询（默认）：依次分配，适合后端服务器性能一致。②权重(weight)：性能好的服务器权重高，分配到更多请求。③IP哈希(ip_hash)：同一客户端IP始终分配到同一台后端（解决Session问题）。④最少连接(least_conn)：分配给当前连接数最少的服务器（适合长连接场景如WebSocket）。

健康检查：max_fails=3 fail_timeout=30s表示30秒内失败3次则标记为down，停用30秒后再试。主动健康检查（需安装nginx_upstream_check_module模块）。

Keepalived高可用：解决Nginx本身的单点故障。两台Nginx服务器（一主一备），共享一个虚拟IP（VIP）。主挂掉后，备机自动接管VIP。Keepalived通过VRRP协议（心跳检测）实现故障切换——主备之间持续发送心跳包，如果备机连续3个心跳周期没收到，就认为自己成了新的master。

如何用
# 负载均衡配置
upstream backend {
    server 192.168.1.10:8080 weight=3;  # 权重3
    server 192.168.1.11:8080 weight=1;
    server 192.168.1.12:8080 backup;    # 备用
}
# Keepalived主配置（/etc/keepalived/keepalived.conf）
vrrp_instance VI_1 {
    state MASTER
    interface eth0
    virtual_router_id 51
    priority 100          # 主100 备90
    virtual_ipaddress { 192.168.1.200 }  # VIP
}

数据库CRUD、WHERE条件、排序分页、聚合函数
达标
详细讲解

CRUD是数据库最核心的四项操作：Create（INSERT）、Read（SELECT）、Update（UPDATE）、Delete（DELETE）。运维不需要像DBA那样精通SQL，但必须能写查询来分析数据、排查问题。

WHERE条件：AND/OR/NOT逻辑运算、BETWEEN ... AND ...区间、IN (...)多值匹配、LIKE '%关键词%'模糊匹配（%任意字符）。

ORDER BY 列名 ASC/DESC排序，LIMIT N OFFSET M分页（跳过M条取N条，等价于LIMIT M,N）。

聚合函数：COUNT(*)计数、SUM()求和、AVG()平均值、MAX()/MIN()最大最小。GROUP BY分组聚合（必须和聚合函数搭配），HAVING过滤分组后的结果（WHERE过滤行，HAVING过滤组）。

如何用
-- 创建表
CREATE TABLE users (
    id INT PRIMARY KEY AUTO_INCREMENT,
    name VARCHAR(50) NOT NULL,
    email VARCHAR(100) UNIQUE,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
-- 复杂查询：按部门统计平均薪资，只显示平均>5000的
SELECT dept, COUNT(*) AS cnt, AVG(salary) AS avg_sal
FROM employees
WHERE status = 'active'
GROUP BY dept
HAVING avg_sal > 5000
ORDER BY avg_sal DESC;

数据类型与约束：数值/字符串/日期时间/主键外键
达标
详细讲解

数值类型：INT整数（4字节）、BIGINT大整数（8字节）、DECIMAL(M,D)精确小数（金融计算必须用这个，FLOAT/DOUBLE有精度问题）、TINYINT微小整数（1字节，常用于布尔值0/1）。

字符串类型：CHAR(N)定长（永远占用N字符，速度快但浪费空间）、VARCHAR(N)变长（实际占用内容长度+1-2字节，省空间但速度略慢）。TEXT大文本（最多64KB）。

日期类型：DATE日期、TIME时间、DATETIME日期时间、TIMESTAMP时间戳（自动更新，范围1970-2038）。

约束保证数据完整性：PRIMARY KEY主键（唯一+非空，建议用自增ID而非业务字段）、FOREIGN KEY外键（保证引用完整性）、UNIQUE唯一、NOT NULL非空、DEFAULT默认值。

如何用
CREATE TABLE orders (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    user_id INT NOT NULL,
    amount DECIMAL(10,2) NOT NULL COMMENT '金额，精确到分',
    status ENUM('pending','paid','shipped','cancelled') DEFAULT 'pending',
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (user_id) REFERENCES users(id)
);

多表关联：JOIN/子查询/事务与隔离级别
达标
详细讲解

JOIN类型：①INNER JOIN（内连接，取两表匹配的行）②LEFT JOIN（左连接，左表全部行+右表匹配行，不匹配为NULL）③RIGHT JOIN（右连接）④CROSS JOIN（笛卡尔积，少用）。LEFT JOIN是运维最常用的——如"查询所有用户及其订单（没有订单的用户也要显示）"。

子查询：在SELECT/WHERE/FROM中嵌套查询。WHERE id IN (SELECT user_id FROM orders WHERE amount > 1000)。子查询可读性差时可以改用JOIN。

事务（Transaction）：一组SQL操作要么全部成功，要么全部回滚。ACID特性：原子性（不可分割）、一致性（数据完整）、隔离性（并发互不干扰）、持久性（提交后不丢失）。START TRANSACTION → SQL操作 → COMMIT/ROLLBACK。

隔离级别（从低到高）：READ UNCOMMITTED（脏读）→ READ COMMITTED（不可重复读，Oracle默认）→ REPEATABLE READ（幻读，MySQL默认）→ SERIALIZABLE（串行）。隔离越高并发越差。

如何用
-- LEFT JOIN：所有用户及其订单
SELECT u.name, o.amount, o.created_at
FROM users u LEFT JOIN orders o ON u.id = o.user_id;
-- 事务转账
START TRANSACTION;
UPDATE accounts SET balance = balance - 100 WHERE id = 1;
UPDATE accounts SET balance = balance + 100 WHERE id = 2;
COMMIT;  -- 或 ROLLBACK 回滚

数据库设计：E-R模型、范式、用户权限管理
达标
详细讲解

E-R模型：实体-关系图。实体（矩形）代表表，属性（椭圆）代表字段，关系（菱形）代表表之间的关联（1:1一对一/1:N一对多/M:N多对多）。画E-R图是数据库设计的第一步，帮助理清业务逻辑。

数据库范式：①1NF：列不可再分（每列都是原子值，不能存JSON数组当一列）。②2NF：非主键列完全依赖主键（不能部分依赖，联合主键时常见问题）。③3NF：非主键列只依赖主键（消除传递依赖）。实际工作中通常满足3NF即可，过度规范化会导致查询需要大量JOIN影响性能。

用户权限管理：CREATE USER 'username'@'host' IDENTIFIED BY 'password';创建用户。host可以是'%'（任意主机）、'192.168.1.%'（网段）、'localhost'（仅本机）。GRANT SELECT,INSERT ON db.table TO 'username'@'host';授予权限。REVOKE回收权限。遵循最小权限原则：只给必要的权限。

如何用
-- 创建只读用户
CREATE USER 'readonly'@'192.168.1.%' IDENTIFIED BY 'Str0ngP@ss';
GRANT SELECT ON mydb.* TO 'readonly'@'192.168.1.%';
-- 创建应用账号（仅CRUD）
CREATE USER 'appuser'@'%' IDENTIFIED BY 'AppP@ss123';
GRANT SELECT, INSERT, UPDATE, DELETE ON mydb.* TO 'appuser'@'%';
FLUSH PRIVILEGES;

第3章 NoSQL：Redis与MongoDB

1
NoSQL概念、分类与对比
达标
详细讲解

NoSQL（Not Only SQL）不是"不用SQL"，而是"不只用关系型"。传统关系型数据库（MySQL/PostgreSQL）适合结构化数据+强一致性需求（如交易系统），但面对海量数据+高并发+灵活Schema的场景时力不从心。

NoSQL四大分类：①键值存储（Redis/Memcached）：极快，适合缓存/Session/计数器。②文档存储（MongoDB）：JSON文档，Schema灵活，适合日志/内容管理/用户画像。③列族存储（HBase/Cassandra）：按列存储，适合大数据分析/时序数据。④图数据库（Neo4j）：存节点和边，适合社交关系/推荐系统。

2
Redis安装、配置文件、客户端工具
达标
详细讲解

Redis（Remote Dictionary Server）是内存型键值数据库，所有数据存在内存中（也支持持久化到磁盘）。单机QPS可达10万+，远高于MySQL。核心数据结构：String、Hash、List、Set、Sorted Set。

为什么Redis快？ ①纯内存操作（磁盘数据库需要寻道和旋转延迟）；②单线程模型（避免多线程上下文切换和锁竞争，Redis 6.0后引入了多线程网络IO）；③高效数据结构（SDS简单动态字符串/SkipList跳表等专门优化的数据结构）。

redis-cli是官方命令行客户端。redis-cli -h 主机 -p 端口 -a 密码连接。键名建议用冒号分隔，如user:1001:profile。

持久化：①RDB（快照）：每隔一段时间把内存数据快照保存到磁盘dump.rdb，恢复快但有数据丢失窗口。②AOF（追加日志）：记录每条写命令，数据安全性高但文件大恢复慢。

如何用
# 安装Redis
dnf install -y redis
systemctl start redis
# 连接与基础操作
redis-cli
SET user:1001:name "张三"
GET user:1001:name
EXPIRE user:1001:name 3600  # 1小时后过期
3
Redis数据类型：String/Hash/List/Set/SortedSet
达标
详细讲解

String：最基本类型。可存字符串/整数/浮点数/二进制。典型场景：缓存JSON数据、分布式锁（SETNX）、计数器（INCR原子递增）。操作：SET/GET/INCR/DECR/SETEX。

Hash：键值对的集合，适合存对象（如用户信息）。优点：可以单独读写某个字段。操作：HSET/HGET/HGETALL/HDEL。

List：双向链表，适合消息队列/最新动态列表。LPUSH头部插入+RPOP尾部弹出=队列（FIFO）。操作：LPUSH/RPUSH/LPOP/RPOP/LRANGE。

Set：无序不重复集合，适合标签系统/共同好友/去重。操作：SADD/SMEMBERS/SINTER（交集）/SUNION（并集）。

Sorted Set：有序集合，每个元素有一个score，按分排序。适合排行榜/优先级队列。操作：ZADD/ZRANGE/ZREVRANGE。

如何用
# Hash - 用户信息
HSET user:1001 name "张三" age 28 city "北京"
HGET user:1001 name
# List - 消息队列
LPUSH queue:tasks "task1" "task2" "task3"
RPOP queue:tasks  # 先进先出
# Sorted Set - 排行榜
ZADD leaderboard 9999 "player1" 8888 "player2"
ZREVRANGE leaderboard 0 9 WITHSCORES

Redis缓存策略、持久化、主从与哨兵集群
达标
详细讲解

缓存三大问题：①缓存穿透：查询不存在的数据，直接打到数据库。解决：布隆过滤器或缓存空值。②缓存击穿：热点key过期瞬间大量请求打到数据库。解决：互斥锁或逻辑过期。③缓存雪崩：大量key同时过期或Redis宕机。解决：过期时间加随机值+集群高可用。

主从复制：一主多从，主写从读，读写分离提升性能。

Sentinel哨兵：监控Redis主从状态，主故障时自动选举新主。至少需要3个哨兵节点避免脑裂。

如何用
# 哨兵配置
sentinel monitor mymaster 192.168.1.10 6379 2
sentinel down-after-milliseconds mymaster 5000

MongoDB安装、CRUD、副本集、聚合管道
达标
详细讲解

MongoDB是文档型数据库，数据以BSON（二进制JSON）格式存储。最大特点：Schema灵活——同一集合中的文档可以有不同的字段。

核心概念：Database≈数据库、Collection≈表、Document≈行。MongoDB没有传统SQL的JOIN，推荐嵌入式文档。

副本集（Replica Set）：一主多从，主节点故障时自动选举新主。最少3节点。

聚合管道（Aggregation Pipeline）：一组阶段依次处理数据：$match过滤→$group分组聚合→$sort排序→$project选择字段。

如何用
# 安装MongoDB

达标
详细讲解

容器是轻量级的虚拟化技术。传统虚拟机（VM）需要运行完整的Guest OS（几GB），容器共享宿主机内核（几MB），启动只需毫秒级。容器解决的核心问题："在我机器上能跑，在你机器上跑不了"——容器把应用+依赖+运行环境打包成一个不可变的镜像，在任何支持Docker的机器上都能一致运行。

镜像(Image) vs 容器(Container)：镜像=类（模板，只读），容器=实例（运行中的镜像，可写层）。一个镜像可以启动多个容器。

容器数据卷、网络模式、资源限制
达标
详细讲解

数据卷(Volume)：容器内的数据默认随容器删除而消失。Volume是宿主机上的目录，挂载到容器内，实现数据持久化。三种方式：①-v /host/path:/container/path绑定挂载（直接映射宿主机目录）②-v volume_name:/container/path命名卷（Docker管理）③tmpfs（内存临时存储）。

网络模式：①bridge（默认，容器间通过虚拟网桥通信，需-p映射端口到宿主机）②host（容器直接使用宿主机网络，性能最好但端口冲突风险）③none（无网络）④container（共享另一个容器的网络命名空间）。

资源限制：--memory=512m限制内存、--cpus=1.5限制CPU核数、--restart=on-failure:3失败重启3次。防止单个容器吃光宿主机资源。

如何用
# 数据卷持久化
docker run -d --name mysql -v mysql/lib/mysql -e MYSQL_ROOT_PASSWORD=xxx mysql:8.0
# 资源限制
docker run -d --name web --memory=512m --cpus=1.0 --restart=always nginx
# 自定义网络
docker network create mynet
docker run -d --name app1 --network mynet myapp

第2章 Dockerfile镜像构建与Compose编排

1
Dockerfile指令：FROM/COPY/ADD/RUN/CMD/ENTRYPOINT
达标
详细讲解

Dockerfile是构建镜像的"配方"。每条指令生成一个只读层（Layer），层可以缓存复用（相同指令不重新执行），这是Docker镜像高效的核心。

核心指令：FROM基础镜像（必须是第一条）、COPY/ADD复制文件到镜像（ADD支持URL下载和tar自动解压，但推荐用COPY更清晰）、RUN在构建时执行命令（安装软件）、WORKDIR设置工作目录、ENV设置环境变量、EXPOSE声明端口、CMD容器启动时默认命令（可被docker run参数覆盖）、ENTRYPOINT容器入口（不可被覆盖，适合固定启动命令）。

CMD vs ENTRYPOINT：ENTRYPOINT定义不可变的启动命令，CMD作为默认参数传给ENTRYPOINT。如ENTRYPOINT ["nginx"] + CMD ["-g","daemon off;"]，运行docker run nginx -t时-t会替换CMD。

如何用
# 多阶段构建Python应用
FROM python:3.11-slim AS builder
WORKDIR /app
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt
FROM python:3.11-slim
WORKDIR /app
COPY --from=builder /usr/local/lib/python3.11/site-packages /usr/local/lib/python3.11/site-packages
COPY . .
EXPOSE 8000
CMD ["python", "app.py"]

镜像构建优化、多阶段构建、私有仓库
达标
详细讲解

构建优化原则：①层缓存：把不常变化的指令放前面（如COPY requirements.txt→RUN pip install），常变化的放后面（如COPY源码）。②多阶段构建：编译阶段用大镜像（含编译器），运行阶段用小镜像（只复制编译产物），最终镜像从几百MB降到几十MB。③.dockerignore：排除不需要的文件（.git/node_modules），减小构建上下文。

私有仓库：企业内网部署Harbor/Docker Registry存储私有镜像。docker tag myapp:1.0 registry.internal:5000/myapp:1.0→docker push。

如何用

docker build -t myapp:1.0 .
docker tag myapp:1.0 registry.internal:5000/myapp:1.0
docker push registry.internal:5000/myapp:1.0
# .dockerignore
echo "node_modules
.git
*.log" > .dockerignore

第3章 Kubernetes基础

1
K8s架构（Master/Node组件）与核心概念
达标
详细讲解

Kubernetes（K8s）是容器编排平台，管理成千上万个容器的部署、扩展、网络、存储。Docker解决了单机容器运行，K8s解决多机容器集群管理。K8s的核心价值：①自愈（容器挂了自动重启、节点挂了自动迁移Pod）②弹性伸缩（流量高峰自动扩容，低谷缩容）③服务发现与负载均衡（不用手动配Nginx）④滚动更新（不中断服务升级）。

Master组件：①kube-apiserver（一切操作的入口，REST API网关）②etcd（分布式KV存储，存集群所有状态）③kube-scheduler（调度器，决定Pod跑在哪个Node上）④kube-controller-manager（控制器管理器，维护期望状态）。

Node组件：①kubelet（节点代理，管理Pod生命周期）②kube-proxy（网络代理，Service负载均衡规则落地）③Container Runtime（容器运行时，如containerd）。

Pod是K8s最小调度单元，一个Pod可以包含1+个紧密耦合的容器（共享网络和存储命名空间）。

如何用
# 安装minikube（本地开发）
minikube start --driver=docker
# 查看集群状态
kubectl cluster-info
kubectl get nodes

kubectl命令行、Pod生命周期
达标
详细讲解

kubectl是K8s的命令行工具。一切操作通过kube-apiserver REST API，kubectl封装了这些API。核心模式：kubectl <动词> <资源类型> [参数]。如kubectl get pods查看Pod列表。

Pod生命周期：Pending（调度中）→ Running（运行中）→ Succeeded（成功退出）/ Failed（失败退出）。Pod不是直接创建的，而是通过更高级别资源（Deployment/StatefulSet/DaemonSet）间接管理。

kubectl核心命令：get查看资源、describe查看详情（排错用）、logs查看Pod日志、exec -it pod -- bash进入容器内部、apply/create创建资源、delete删除资源、scale扩缩容、rollout管理更新。

如何用

第4章 Kubernetes网络与存储

1
Service四层负载均衡（ClusterIP/NodePort/LoadBalancer）
达标
详细讲解

Service是K8s的"内置负载均衡器"。Pod IP是动态的（重启就变），Service提供一个固定的虚拟IP（ClusterIP）和DNS名，自动将流量转发到匹配的Pod。解决了"Pod IP变化后其他服务如何找到它"的问题。

三种Service类型：①ClusterIP（默认）：集群内部IP，只能在集群内访问。②NodePort：在每个Node上开一个端口（30000-32767），外部可通过NodeIP:NodePort访问。③LoadBalancer：云服务商提供外部LB（如阿里云SLB），从LB转发到NodePort。

如何用
apiVersion: v1
kind: Service
metadata: {name: nginx-svc}
spec:
  type: NodePort
  selector: {app: nginx}
  ports:
  - port: 80
    targetPort: 80
    nodePort: 30080
# 访问：NodeIP:30080

Ingress七层路由与Ingress Controller
达标
详细讲解

Ingress是K8s的七层（HTTP/HTTPS）路由。Service只能做简单的四层转发，Ingress可以根据域名、URL路径、Header等做复杂的七层转发（类似Nginx）。比如api.example.com/v1→Service A，api.example.com/v2→Service B。

Ingress Controller是Ingress规则的执行者（如Nginx Ingress Controller、Traefik），本身也运行在K8s中。

如何用

      paths:
      - path: /api
        pathType: Prefix
        backend: {service: {name: api-svc, port: {number: 80}}}

PV/PVC/StorageClass持久化存储
达标
详细讲解

K8s存储体系的三层抽象：PV（PersistentVolume，管理员创建的存储池，如NFS/云盘）→PVC（PersistentVolumeClaim，用户申请存储，类似"我要5G读写一次的存储"）→StorageClass（动态创建PV，用户只需PVC无需手动创建PV）。

容器重启后数据丢失，Pod挂载PVC实现数据持久化。StatefulSet常用PVC为每个Pod创建独立存储。

如何用
# PVC示例
apiVersion: v1
kind: PersistentVolumeClaim
metadata: {name: mysql-pvc}
spec:
  accessModes: [ReadWriteOnce]
  resources: {requests: {storage: 10Gi}}
# Pod引用
volumes:
- name: mysql-data
  persistentVolumeClaim: {claimName: mysql-pvc}

第5章 Kubernetes核心控制器

1
Deployment部署、滚动更新、回滚、扩缩容
达标
详细讲解

Deployment是最常用的控制器，管理无状态应用（如Web服务）。它的核心能力：①ReplicaSet管理：保证指定数量的Pod副本始终运行（死一个自动补一个）；②滚动更新：逐步替换旧版本Pod为新版本（先启动新的→再停旧的），零停机升级；③回滚：更新出问题时一键回滚到上一版本；④扩缩容：一行命令增加/减少副本数。

滚动更新策略：maxSurge: 1（更新时最多多1个Pod）、maxUnavailable: 1（最多有1个Pod不可用）。生产环境谨慎设置避免服务中断。

如何用
# 创建Deployment
kubectl create deployment web --image=nginx:1.24 --replicas=3
# 滚动更新
kubectl set image deployment/web nginx=nginx:1.25
kubectl rollout status deployment/web  # 查看更新进度
# 回滚
kubectl rollout undo deployment/web
# 扩缩容
kubectl scale deployment web --replicas=5
2
StatefulSet有状态应用管理（ZooKeeper/MQ）
达标
详细讲解

StatefulSet用于管理有状态应用（如数据库/消息队列/分布式存储）。与Deployment的区别：①Pod名字固定（如mysql-0, mysql-1），重启后名字不变；②每个Pod独立存储（通过volumeClaimTemplates）；③有序部署和缩容（先启动mysql-0再mysql-1，缩容反过来）。

为什么数据库不能用Deployment？因为数据库需要稳定的网络标识+持久化存储+有序更新，Deployment的Pod是匿名的（随机名字+共享存储）。

如何用
apiVersion: apps/v1
kind: StatefulSet
metadata: {name: zk}
spec:
  serviceName: zk
  replicas: 3
  selector: {matchLabels: {app: zk}}
  template:
    metadata: {labels: {app: zk}}
    spec:
      containers:
      - name: zookeeper
        image: zookeeper:3.8
  volumeClaimTemplates:
  - metadata: {name: data}
    spec:
      accessModes: [ReadWriteOnce]
      resources: {requests: {storage: 10Gi}}

RBAC权限控制、ServiceAccount、安全上下文
达标
详细讲解

RBAC（基于角色的访问控制）是K8s的权限体系：Subject（谁）→ Role/ClusterRole（能干什么）→ RoleBinding/ClusterRoleBinding（绑定）。Role只在Namespace内生效，ClusterRole跨Namespace。

ServiceAccount是Pod的"身份证"，Pod通过SA与API Server通信。Pod安全上下文：控制容器运行时的权限（如是否特权容器、文件系统只读、以非root用户运行）。

如何用
# 创建只读Role
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata: {namespace: default, name: pod-reader}
rules:
- apiGroups: [""]
  resources: ["pods"]
  verbs: ["get", "list"]
# 绑定到用户
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata: {name: read-pods, namespace: default}
subjects: [{kind: User, name: "zhangsan", apiGroup: rbac.authorization.k8s.io}]
roleRef: {kind: Role, name: pod-reader, apiGroup: rbac.authorization.k8s.io}

达标

Pipeline流水线脚本（Declarative Pipeline）
达标
详细讲解

Declarative Pipeline是Jenkins推荐的流水线写法（相比Scripted Pipeline更简洁）。结构：pipeline{ agent{...} stages{ stage('Build'){ steps{...} } stage('Test'){...} stage('Deploy'){...} } }。

每个stage代表一个阶段，所有stage都成功流水线才成功。常见流水线阶段：Checkout（拉代码）→ Build（编译/构建镜像）→ Test（单元测试/集成测试）→ Security Scan（安全扫描）→ Deploy Staging（部署测试环境）→ Deploy Production（部署生产，需人工审批）。

如何用

    agent any
    environment {
        DOCKER_REGISTRY = 'registry.internal:5000'

            steps { git 'https://gitlab.com/myapp.git' }
        }
        stage('Build Image') {
            steps { sh 'docker build -t $DOCKER_REGISTRY/myapp:$BUILD_NUMBER .' }
        }
        stage('Push') {
            steps { sh 'docker push $DOCKER_REGISTRY/myapp:$BUILD_NUMBER' }
        }
        stage('Deploy to K8s') {
            steps { sh 'kubectl set image deployment/myapp myapp=$DOCKER_REGISTRY/myapp:$BUILD_NUMBER' }
        }
    }
    post {
        failure { emailext to:'ops@company.com', subject:"构建失败" }
    }
}

达标

达标

达标
详细讲解

output {

}

第3章 可观测性体系（OpenTelemetry/Jaeger/Grafana）

1
可观测性三支柱：Metrics/Logs/Traces
达标
详细讲解

可观测性（Observability）是比传统监控更高级的概念。三个支柱：

①Metrics（指标）：数值型数据，回答"系统有什么问题？"——CPU 90%、请求延迟 500ms。Prometheus/Zabbix是代表。

②Logs（日志）：文本型数据，回答"为什么出问题？"——"NullPointerException at line 42"。ELK是代表。

③Traces（链路追踪）：请求在微服务间的完整路径，回答"问题出在哪个环节？"——请求经过API Gateway→Auth Service→Order Service→DB，Order Service耗时3秒。Jaeger/Zipkin是代表。

三支柱关联：一条Trace ID可以串起所有相关Log和Metrics，实现"发现异常→定位到哪个服务→查看关联日志"的完整排错链。

Jaeger是分布式追踪后端（类似ES之于日志），用于存储和可视化Trace数据。核心概念：Span：一个操作单元（一次HTTP调用/一次数据库查询）、Trace：多个Span组成一个完整请求链路。Jaeger UI可以展示调用拓扑和水瀑布图。

幂等性（Idempotency）：Ansible的核心特性——重复执行同一任务，结果一致（不会重复安装已安装的软件）。这是自动化运维安全性的保证。

Playbook是Ansible的核心，YAML格式定义一系列任务（task）。一个Playbook可以包含多个Play（每个Play针对一组主机）。

Roles：Playbook的模块化组织方式。一个Role包含：tasks（任务）、handlers（触发器，如配置变更后重启服务）、templates（Jinja2模板）、vars（变量）、defaults（默认变量）、files（静态文件）。Roles让复杂的自动化任务变得可复用、可维护。

Jinja2模板：动态生成配置文件。如Nginx配置模板中worker_processes {{ ansible_processor_vcpus }};根据目标机器CPU核数自动设置。

如何用

  become: yes
  vars:
    nginx_port: 80

  - name: Install Nginx
    yum: {name: nginx, state: present}
  - name: Copy config
    template: {src: nginx.conf.j2, dest: /etc/nginx/nginx.conf}
    notify: Restart Nginx
  - name: Start Nginx
    service: {name: nginx, state: started, enabled: yes}
  handlers:
  - name: Restart Nginx
    service: {name: nginx, state: restarted}

达标
详细讲解

terraform {

provider "alicloud" {
  region = "cn-hangzhou"
}
resource "alicloud_instance" "web" {
  instance_name = "web-server"
  image_id      = "ubuntu_22_04_x64"
  instance_type = "ecs.g6.large"
  vswitch_id    = "vsw-xxxxx"
}

系统调优（sysctl/kernel参数）
良好
详细讲解

# 查看所有参数
sysctl -a | grep tcp

第3章 故障排查方法论与实战

1
故障排查方法论（USE/RED/GREP方法）
达标
详细讲解

USE方法（资源分析）：对每种资源（CPU/内存/磁盘/网络），检查三个维度：Utilization（利用率）、Saturation（饱和度，队列长度）、Errors（错误）。适合系统资源瓶颈排查。

RED方法（微服务分析）：对每种请求（服务），检查：Rate（请求率）、Errors（错误率）、Duration（延迟）。适合微服务性能分析。

故障排查黄金法则：①先复现（能复现的问题才能解决）②二分法（逐步缩小范围）③变更关联（最近改了什么？）④数据说话（用监控数据而非猜测）。

2
典型故障案例：OOM/死锁/网络分区/雪崩
达标
详细讲解

OOM（Out of Memory）：内存耗尽，进程被OOM Killer杀掉。排查：dmesg | grep -i oom看被杀进程、free -h看内存使用、分析内存泄漏（Java用jmap/jstat，Python用memory_profiler）。

死锁：多个进程互相等待对方释放资源。排查：jstack看Java线程栈、lsof看文件锁、strace跟踪系统调用。

网络分区：网络中断导致集群节点间无法通信。排查：ping/traceroute测连通性、tcpdump抓包、检查防火墙规则。

雪崩：一个服务故障导致依赖它的服务连锁故障。排查：看调用链（Jaeger）、限流/熔断/降级是否生效、依赖服务健康状态。

如何用
# OOM排查
dmesg | grep -i "out of memory"
# 死锁排查（Java）
jstack  | grep -A 20 "BLOCKED"
# 网络分区排查
ping -c 5 target-host
traceroute target-host
tcpdump -i eth0 host target-ip -nn

详细讲解

Function Calling让AI模型能决定调用哪些函数。在运维中，可以定义一组运维工具函数：get_server_status(hostname)、restart_service(service_name)、query_logs(service, time_range)、send_alert(message, level)。

AI收到"server-01的nginx出问题了"→自动调用get_server_status("server-01")→发现nginx服务停止→调用restart_service("nginx")→确认重启成功→调用query_logs("nginx","last 10min")分析根因。

如何用
tools = [{

    "function": {

        }}
    }
